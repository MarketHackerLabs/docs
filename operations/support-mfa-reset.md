# Сброс MFA: инструкция для поддержки

Сброс MFA нужен, когда пользователь потерял телефон / приложение-аутентификатор
**и** не может воспользоваться резервными кодами. Действие необратимо с точки
зрения второго фактора: после сброса достаточно знать пароль (или получить новый
через почту).

**Сначала спросите про резервные коды** (Настройки → Безопасность при включении
MFA пользователь должен был их сохранить). Если код есть — пусть войдёт сам,
без сброса в админке.

Если кодов нет / все израсходованы — **подтверждаем владение аккаунтом**, и
только потом жмём «Сбросить MFA» (право `users:mfa:reset`).

См. также: [Восстановление пароля](../integrations/password-recovery.md),
[Аутентификация](../architecture/authentication.md).

---

## Что считается доказательством владения

Идентификатор аккаунта в MarketHacker — **email**. Кто контролирует этот ящик,
может сбросить пароль сам. Сброс MFA без проверки почты опасен: злоумышленник
с украденным паролем мог бы попросить поддержку «просто отключить MFA».

| Уровень | Что проверить | Зачем |
|--------|----------------|------|
| **Обязательно** | Контроль почты аккаунта | Главный доказательный фактор |
| **Обязательно** | Пароль известен **или** пароль уже сброшен через письмо на этот email | Отсекает «я просто знаю чужой email» |
| **Желательно** | Факты из карточки (организация, кабинет, дата регистрации) | Доп. защита от социальной инженерии |

Резервные MFA-коды — самообслуживание пользователя. Поддержка их не выдаёт и не
видит (в БД только хеши). Опираемся на почту и данные аккаунта.

## Когда можно сбрасывать

Сценарий подходит, если пользователь:

1. Пишет в поддержку (чат или письмо) и указывает email аккаунта.
2. Подтвердил контроль этой почты (см. ниже).
3. Либо входит до шага MFA (пароль верный, код недоступен), либо уже сбросил
   пароль через «Забыли пароль?» на тот же email.

**Не сбрасывать**, если:

- пишет «с чужой» почты / мессенджера и не может подтвердить ящик аккаунта;
- просит сбросить MFA коллеге / «директору», не будучи владельцем и без явного
  подтверждения с почты владельца аккаунта;
- не совпадают базовые факты по карточке и нет контроля почты;
- аккаунт заблокирован — сначала разобраться со статусом (с суперадмином).

## Пошаговый протокол

### 1. Найти пользователя

Админка → **Пользователи** → карточка по email из обращения.  
Проверить: аккаунт активен, MFA действительно включена, список организаций.

### 2. Подтвердить контроль почты (обязательно)

Выбрать **один** способ:

**А. Письмо со стороны пользователя**  
Обращение пришло **с того же адреса**, что в карточке (полное совпадение, не
похожий домен). В чате поддержки email клиента должен совпадать с карточкой.

**Б. Проверка «получили письмо?»** (если пишут не с ящика аккаунта)  
1. Нажать в карточке **«Сбросить пароль»** — уйдёт ссылка на email аккаунта  
   **или** попросить пользователя самому нажать «Забыли пароль?» на портале.  
2. Попросить в ответе назвать **последние 6 символов** ссылки из письма  
   (или точную тему/время письма, если ссылку светить нельзя).  
3. Совпадение → почта под контролем.

**В. Уже вошёл по паролю до MFA**  
Пользователь пишет из чата после частичного входа / описывает экран «введите код
из приложения». Тогда пароль уже проверен системой; всё равно желательно
подтвердить почту способом Б, если канал обращения сомнительный.

### 3. Дополнительная сверка (желательно, особенно на платных / owner-аккаунтах)

Спросить 1–2 факта и сверить с карточкой, **не подсказывая ответ**:

- название организации или кабинета WB/Ozon;
- роль (владелец / участник);
- примерно, когда регистрировался или последний раз заходил.

При серьёзных расхождениях — не сбрасывать, эскалировать.

### 4. Сбросить MFA

1. В карточке: **Сбросить MFA**.  
2. Система отключит MFA и отзовёт все сессии.  
3. В чате поддержки кратко зафиксировать:  
   `MFA reset: email OK (способ А/Б/В), org=…, agent=…`.

### 5. Что сказать пользователю после сброса

1. Войти на портале с паролем (без кода MFA).  
2. Сразу включить MFA снова в **Настройки → Безопасность**.  
3. Сохранить доступ к приложению-аутентификатору (или завести второй фактор
   на запасном устройстве).  
4. Если пароль тоже подозревают скомпрометированным — сменить пароль после входа.

## Типовые кейсы

| Ситуация | Действие |
|----------|----------|
| Есть резервный код | Пользователь входит сам кодом на шаге MFA; сброс не нужен |
| Потерял телефон, пароль помнит, пишет с email аккаунта | Сверка фактов → сброс MFA |
| Потерял телефон и пароль | Сначала сброс пароля через почту → убедиться, что вошёл / получил ссылку → сброс MFA |
| Просит сбросить MFA «другу» | Только с почты владельца аккаунта по протоколу выше |
| Подозрение на взлом | Не сбрасывать MFA по первому запросу; сброс пароля на email + эскалация; при необходимости заблокировать аккаунт (суперадмин) |

## Права и аудит

| Действие | Право |
|----------|--------|
| Открыть карточку | `users:read` |
| Письмо со ссылкой сброса пароля | `users:password:reset` |
| Сброс MFA | `users:mfa:reset` |

Событие в audit log: `admin.user.mfa_reset`.  
Права сотрудникам: **Поддержка → Сотрудники**.
